In the PATCH handler, locate the notification insert block (lines 44-55 in current file). Change the `.insert()` call to `.upsert()` with `onConflict: "user_id,event_id,type"`.

The upsert should NOT use `ignoreDuplicates: true` (unlike the cron route pattern). Instead, it should overwrite `read`, `created_at`, `title`, and `message` so the organizer sees a fresh, unread notification when their event status changes again (e.g., reject then re-approve).

Change from:
```typescript
await serviceClient.from("notifications").insert({
  user_id: event.created_by,
  type: isApproved ? "event_approved" : "event_rejected",
  title: isApproved ? "Event Approved!" : "Event Not Approved",
  message: isApproved
    ? `Your event "${event.title}" has been approved and is now live.`
    : `Your event "${event.title}" was not approved.`,
  event_id: id,
});
```

Change to:
```typescript
await serviceClient.from("notifications").upsert(
  {
    user_id: event.created_by,
    type: isApproved ? "event_approved" : "event_rejected",
    title: isApproved ? "Event Approved!" : "Event Not Approved",
    message: isApproved
      ? `Your event "${event.title}" has been approved and is now live.`
      : `Your event "${event.title}" was not approved.`,
    event_id: id,
    read: false,
    created_at: new Date().toISOString(),
  },
  { onConflict: "user_id,event_id,type" }
);
```

Key differences from the insert:
- Uses `.upsert()` instead of `.insert()`
- Adds `onConflict: "user_id,event_id,type"` to match the notifications_dedup_idx partial unique index
- Adds `read: false` so re-approve/re-reject surfaces as unread
- Adds `created_at: new Date().toISOString()` so the notification appears fresh

Do NOT modify any other part of the route — the status update logic, admin verification, error handling, and try/catch structure all stay the same.
